# Cofind 2

Cofind 2 - fullstack-платформа для поиска соавторов, соигроков, ролевых партнеров и творческих команд.

## Что уже есть

- `apps/web` - интерактивный фронтенд-прототип сайта.
- `apps/api` - Prisma-домен Cofind 2: пользователи, профили, заявки, чаты, реакции, цитаты, холст, жалобы, баны, реклама, подписки, SEO и audit log.
- `docker-compose.yml` - Postgres, Redis и Meilisearch для локальной разработки.
- `deploy/docker-compose.prod.example.yml` - пример production-compose для API, web, Postgres, Meilisearch и volume загрузок.
- `deploy/docker-compose.cofind2.yml` - production-compose для `cofind2.com` с Caddy, автоматическим HTTPS, web/API/WS на одном домене.
- `deploy/.env.production.example` - шаблон production-переменных; реальные секреты генерируются через `pnpm secrets:generate` и не должны попадать в репозиторий.
- `deploy/.env.production.cofind2.example` - доменный шаблон production-переменных для `https://cofind2.com`.
- `deploy/nginx.reverse-proxy.example.conf` - пример внешнего Nginx reverse proxy с HTTPS, API и WebSocket `/ws/chat`.
- `deploy/SERVER_CHECKLIST.md` - пошаговый чеклист переноса на сервер и проверки после запуска.
- `MVP_CHECKLIST.md` - короткий чеклист запуска, smoke-проверок и ручной приемки MVP.

## Быстрый запуск фронтенда

```bash
pnpm install
pnpm --filter @cofind/web dev
```

По умолчанию сайт открывается на `http://localhost:3000`. Уже запущенный вручную сервер можно поднять на другом порту:

```bash
$env:PORT="4173"; pnpm --filter @cofind/web dev
```

Фронтенд сначала пытается подключиться к API `http://localhost:4000/api/v1`. Если API недоступен, сайт автоматически остается в mock-режиме, поэтому его можно открывать без базы данных. Чтобы указать другой API:

```js
localStorage.setItem("cofindApiBase", "http://localhost:4000/api/v1")
```

Для деплоя без ручной правки JS web build принимает публичные URL:

```powershell
$env:PUBLIC_WEB_URL="https://cofind.example.com"
$env:PUBLIC_API_BASE="https://api.cofind.example.com/api/v1"
pnpm --filter @cofind/web build
```

Сборка обновит `dist/index.html`, `robots.txt` и `sitemap.xml` под выбранный домен.

## База данных

```bash
copy .env.example .env
docker compose up -d postgres redis meilisearch
pnpm --filter @cofind/api prisma:migrate --name init
pnpm --filter @cofind/api seed
```

For production migrations use:

```bash
pnpm --filter @cofind/api prisma:deploy
```

## API

```bash
pnpm --filter @cofind/api dev
```

Для текущего локального dev-окружения с Postgres на `5433` и Meilisearch на `7700` можно запустить API одной командой:

```powershell
pnpm api:dev:local
```

Helper выставляет обе пары переменных Meilisearch: `MEILISEARCH_*` и совместимые `MEILI_*`.

После запуска:

- REST API: `http://localhost:4000/api/v1`
- Swagger: `http://localhost:4000/api/docs`
- Liveness: `GET /api/v1/health` или `GET /api/v1/health/live`
- Readiness: `GET /api/v1/health/ready` проверяет Postgres (`SELECT 1`), Meilisearch (`/health`) и realtime-компонент чата; отдаёт `200` когда всё ок и **HTTP 503** если хоть одна зависимость недоступна (мониторинг ловит по коду). `dependencies.realtime` включает `metrics` (счётчики WS: `connections`, `disconnects`, `errorFrames`, `socketErrors`, `droppedBackpressure`, `rejectedOverload`).

### Наблюдаемость

Подробности и пороги алертов — в [docs/observability.md](docs/observability.md).

- **Sentry (фронт + бэк), полностью DSN-driven:** без `SENTRY_DSN` SDK не инициализируется (no-op). API (`@sentry/node`) репортит 5xx с тегом `request_id`. Фронт-SDK самохостится (`apps/web/vendor/sentry.min.js`) — CSP `script-src 'self'` запрещает CDN — и включается через `<meta>`, которые `build.js` вставляет при заданном `SENTRY_DSN` (build-arg web). Релиз = git sha из CI: job `build` создаёт релиз и грузит sourcemaps (секреты `SENTRY_AUTH_TOKEN`/`SENTRY_ORG`/`SENTRY_PROJECT`, `SENTRY_URL` для self-hosted), job `deploy` пробрасывает тот же `SENTRY_RELEASE` в сборку на сервере.
- **Request-id + структурные логи:** каждый запрос получает `X-Request-Id` (входящий проксируется, иначе UUID), он же в ответе и в теле 5xx (`requestId`). На каждый ответ — одна JSON-строка в stdout (`http_request`: method, path, status, `durationMs`, requestId, ip); `level` = error/warn/info по коду; liveness-пинги из логов исключены.

Для публичного runtime API подхватывает ближайший `.env`, если переменная еще не задана окружением, и проверяет обязательные значения при старте. В `NODE_ENV=production` нужны сильные разные `JWT_ACCESS_SECRET`/`JWT_REFRESH_SECRET`, `DATABASE_URL`, `MEILISEARCH_HOST`, `MEILISEARCH_MASTER_KEY`, `PUBLIC_WEB_URL`, `PUBLIC_API_BASE`, `PAYMENT_WEBHOOK_SECRET` и `MAIL_WEBHOOK_URL`. Swagger в production выключен по умолчанию; включается только `API_DOCS_ENABLED=true`. Если API стоит за reverse proxy, выставьте `TRUST_PROXY=true`, чтобы rate limit видел реальный клиентский IP.

Сейчас реализован первый API-слой:

- Auth и профиль: `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`, `POST /auth/password-reset/request`, `POST /auth/password-reset/confirm`, `POST /auth/change-password`, `POST /auth/deactivate`, `GET /auth/me`, `PATCH /me/profile`; web-студия умеет редактировать профиль, обложку, социальные ссылки, расширенные предпочтения автора, менять пароль и деактивировать аккаунт с подтверждением пароля.
- Профиль хранит `avatarUrl` и `coverImageUrl`: web-студия позволяет выбрать preset-аватар, загрузить небольшую картинку до 128KB как avatar и загрузить обложку до 256KB; web заранее валидирует PNG/JPEG/WebP и размер файла до чтения/отправки, аватар отображается в шапке, ЛК, публичном профиле, inbox, личных сообщениях и общем чате, а обложка - в ЛК и публичном профиле.
- Перед отправкой avatar, cover, background и рисунков мини-холста web сжимает изображения через canvas до целевых размеров/лимитов, использует lazy/async decoding для динамических `<img>` и стабилизирует размеры рекламных/рисуночных превью.
- Клиент и API принимают для аватаров/обложек/фонов только preset-аватары, `http(s)` URL или `data:image` PNG/JPEG/WebP; опасные схемы вроде `javascript:` и `ftp:` отклоняются до сохранения или не рендерятся.
- Загруженные изображения можно сохранять через `POST /uploads/images`: API принимает JSON/base64 payload до 512KB, пишет PNG/JPEG/WebP до 256KB в локальное хранилище и отдает публичный URL `/uploads/images/:fileName` от `PUBLIC_API_BASE`/`PUBLIC_API_URL`; web-загрузка аватара, обложки и рисунки мини-холста используют этот endpoint при доступном API.
- API удаляет старые локальные файлы `/uploads/images/...` при замене avatar, cover и фонового изображения студии, при деактивации профиля, а также при удалении сообщения общего чата с рисунком мини-холста; внешние URL и data URL не удаляются.
- Web хранит access/refresh token и автоматически обновляет access token при 401, если refresh token еще действителен.
- Восстановление пароля в production отправляет письмо через provider-agnostic `MAIL_WEBHOOK_URL`; ссылка открывает `/auth?resetToken=...` и подставляет token в форму смены пароля. В dev token по-прежнему возвращается в ответе API для smoke.
- Перед подключением realtime-чата web также пробует обновить access token, чтобы reconnect после долгой сессии не падал на истекшем JWT.
- Регистрация проверяет email и username без учета регистра, чтобы дубли не проваливались в ошибку уникального индекса базы.
- Внешний вид: `GET/PATCH /me/preferences`, `POST /me/background`; web-мастерская после входа сохраняет настройки автоматически.
- Web-мастерская внешнего вида умеет сохранять и убирать фоновое изображение студии через `/me/background`; файл PNG/JPEG/WebP до 256KB можно загрузить через `/uploads/images` с `purpose: "background"` и сохранить публичным URL, а удаление фона чистит старый локальный файл.
- Личная студия `/me` после входа загружает профиль, опубликованные заявки, отклики, Premium-статус и уведомления; уведомления можно отмечать прочитанными.
- Личная студия показывает чеклист готовности профиля: аватар, обложка, био, творческие метки, контакты, первая заявка и первый контакт, с быстрыми действиями к нужным разделам.
- Из личной студии можно открыть собственный публичный профиль и скопировать прямую ссылку `/profile/:username`.
- Экспорт данных: `GET /me/export` собирает профиль, настройки, заявки, отклики, диалоги, сообщения, лайки, уведомления, платежи, жалобы и блокировки без password hash; в ЛК есть кнопка скачать JSON.
- Деактивация аккаунта: `POST /auth/deactivate` проверяет текущий пароль, переводит пользователя в `DELETED`, скрывает профиль и заявки, отменяет активную подписку и завершает web-сессию.
- `/me/liked-listings` возвращает понравившиеся публичные заявки пользователя; web-ЛК показывает их отдельным блоком с поиском/сортировкой и позволяет убрать из списка повторным лайком.
- Заявки: `GET/POST /listings`, публичный `GET /listings/:slugOrId` только для опубликованных approved-заявок, `GET /listings/mine`, `GET /listings/mine/:id`, `PATCH /listings/:id`, публикация, закрытие, архив, мягкое удаление владельцем, отклики и лайки.
- SSR-страница заявки `/listings/:slug`: nginx проксирует её на API (`GET /listings/:slug/page`), который сервер-сайд рендерит карточку с уникальными `title`/`description`/`canonical`/OG и JSON-LD `CreativeWork`; неизвестный или снятый с публикации slug отдаёт честный HTTP 404 + `noindex`. Внутри SPA `/listings/:slug` — это полноценный интерактивный экран, карточки ленты/главной/похожих ссылаются на этот канонический URL.
- Динамическая OG-картинка заявки: `og:image` указывает на `/listings/:slug/og.png` (nginx проксирует на API), где satori (HTML/CSS → SVG) + `@resvg/resvg-js` (SVG → PNG) рендерят карточку 1200×630 с заголовком заявки, типом, рейтингом, тегами и брендовыми цветами (`#2fbf9f`), шрифт Roboto (Cyrillic) в `apps/api/assets/fonts`. Результат кешируется на диск по `slug`+`updatedAt` (правка заявки инвалидирует кэш), отдаётся с `Content-Type: image/png` и `Cache-Control: public, max-age=86400`. На любой ошибке роут отдаёт 302 на статичный `/og-image.png`, чтобы соцсети никогда не видели битый `og:image`. В `<head>` добавлены `og:image:width`/`og:image:height` = 1200×630.
- Первая страница ленты `/feed` рендерится на сервере: nginx через SSI подставляет HTML-фрагмент карточек (`GET /api/v1/listings/feed-cards`, проксируется как internal `/_feed_cards`) в статическую SPA-оболочку, со ссылками на `/listings/<slug>` — карточки видны в HTML до JS. Дальше SPA гидрируется поверх (фильтры, сортировка, пагинация); до прихода данных API клиент не затирает серверные карточки (маркер `data-ssr-feed`).
- SSR-страницы каталога для органического трафика: индексы `/fandoms`, `/genres`, `/tags` и страницы сущностей `/fandoms/<slug>`, `/genres/<slug>`, `/tags/<slug>` (nginx проксирует на `GET /api/v1/catalog/:kind[/:slug]/page`). Страница сущности содержит описание, серверно-отрендеренную ленту заявок по ней (со ссылками на `/listings/<slug>` и CTA в интерактивную `/feed?<kind>=<slug>`), перелинковку на соседние сущности и индекс. `title`/`description` берутся из `seoTitle`/`seoDescription` с fallback на шаблон («Поиск соавтора и партнёра по фандому X — Cofind 2»), есть JSON-LD `BreadcrumbList`/`CollectionPage`, неизвестный slug — честный 404 + `noindex`. Все индексы и сущности добавлены в динамический sitemap с `lastmod`.
- Русская плюрализация счётчиков (заявки, отклики, лайки, фандомы) вынесена в утилиту `pluralize(n, [одна, две, пять])` с корректной обработкой 11–14 и переноса на *21/*121: на сервере `apps/api/src/common/pluralize.ts` (SSR-страницы и SEO-тексты — `descTpl` каталога, профиль, карточки), на клиенте — зеркальная копия в `app.js`. Юнит-тест `pnpm test:unit` покрывает 1, 2, 5, 11, 21, 111, 121.
- Единые header и footer на всех страницах: SSR-страницы (`/listings/*`, `/fandoms/*`, `/profile/*`, 404) используют ту же навигацию (Заявки, Как это работает, Правила, Чат) и CTA (Войти, Создать заявку), что и гостевая шапка SPA, и тот же `site-footer`. SPA-страницы (`/`, `/feed`, `/chat`, …) уже имеют уникальные `description`/`og:title`/`og:description` и `og:image` (генерируются в `build.js`). Превью мини-холста в `/chat` не оставляет `<img>` с пустым `src` (картинка создаётся только при наличии рисунка и удаляется при сбросе).
- `GET /listings` всегда пагинирован offset-режимом и отвечает конвертом `{ items, total, page, pageSize, totalPages, nextPage }` (где `nextPage` — номер следующей страницы или `null`). Размер страницы — параметр `pageSize` (или алиас `limit`), по умолчанию 20, клампится в диапазон `[1, 50]`; значения выше 50 кэпируются, а не отклоняются. Фильтры `type`/`ageRating`/`tag`/`genre`/`fandom`/`character`/`q` применяются до пагинации. Лента (`/search/listings`) и каталожные эндпоинты сохраняют свой формат `hits + pagination`; фронт ленты понимает оба конверта.
- Отклики: `GET /listings/mine/responses`, `GET /listings/mine/incoming-responses`, `POST /listings/responses/:id/status`; отправка отклика разрешена только на опубликованную approved-заявку, не на свою и не при блокировке между пользователями.
- Статус отклика после принятия или отклонения терминален, чтобы не плодить повторные уведомления и диалоги.
- Публичные заявки, search/reindex и отклики скрывают авторов со статусом banned/temp-banned/deleted.
- Повторный отклик на ту же заявку от того же пользователя возвращает явный 400, а лайк несуществующей/непубличной заявки возвращает 404.
- При принятии или отклонении отклика отправитель получает уведомление в `/notifications`.
- Публичные профили: `GET /profiles/:username`; профили banned/temp-banned/deleted пользователей скрываются как 404.
- Последняя активность: защищенные запросы обновляют `lastSeenAt` не чаще раза в 5 минут; публичный профиль и админка показывают “последняя активность”.
- Приватность профиля: пользователь может скрыть `lastSeenAt` из публичного профиля и запретить новые личные диалоги с кнопки “Написать”; существующие диалоги не ломаются.
- Социальные ссылки профиля сохраняются в `socialLinks`: сайт/портфолио, Telegram и Discord редактируются в ЛК и показываются в публичном профиле автора.
- Личные диалоги: `POST /conversations`, `GET /conversations`, `GET/POST /conversations/:id/messages`, `DELETE /conversations/:id/messages/:messageId`, `POST /conversations/:id/read`; создание диалога и отправка сообщений учитывают блокировки между участниками и недоступность banned/temp-banned/deleted пользователей.
- `GET /conversations/:id/messages` отдает последние сообщения, а `?cursor=<oldestMessageId>` дозагружает более раннюю историю личного диалога.
- Создание личного диалога с несуществующим участником возвращает 404.
- DTO личных диалогов trim/dedupe'ят participantIds, ограничивают размер группового диалога и не создают лишних дублей участников.
- `/listings/mine`, `/listings/mine/responses`, `/listings/mine/incoming-responses` и `/listings/:id/responses` поддерживают совместимый paged-режим `page/pageSize`.
- Каталог: `GET /tags`, `/genres`, `/fandoms`, `/characters`, `POST /suggestions`.
- Предложения каталога защищены от активных дублей одного автора по type/title.
- Общий чат: сообщения, цитаты, реакции, лайки с сохраненными счетчиками в истории, рисунки мини-холста, удаление своего сообщения; лайки/реакции/цитаты недоступны для удаленных сообщений.
- Сообщения общего чата с `drawingUrl` можно отправлять без отдельного текста: API подставляет безопасную подпись “Отправлен рисунок с мини-холста” и сохраняет изображение.
- Web-composer общего чата показывает лимит 4000 символов, статус отправки через WS/REST и не отправляет пустое сообщение без рисунка.
- Web-чат поддерживает локальный поиск по автору, тексту, роли и цитате, показывает количество найденных сообщений и быстрый сброс поиска.
- Web-чат умеет дозагружать более раннюю историю через cursor `/chat/messages?cursor=...`, добавляя старые сообщения сверху без сброса текущей позиции.
- Web-чат поддерживает комнаты `# общий`, `# поиск соигроков`, `# фандомы`, `# модерация`: выбор сохраняется локально, восстанавливается из URL `/chat?room=...`, прямую ссылку на комнату можно скопировать, а API хранит комнату отдельным полем `room`.
- Лайки заявок и чата, а также emoji-реакции работают как toggle: один пользователь ставит одну отметку и повторным кликом снимает ее; для сообщения активна только одна emoji-реакция пользователя, новая emoji заменяет старую, API возвращает актуальные счетчики после каждого toggle.
- Публичные `GET /listings`, `GET /listings/:slugOrId`, `GET /chat/messages` и `GET /search/listings` остаются доступными без входа, но при Bearer-токене добавляют viewer-state: `likedByMe` для заявок/сообщений и `reactedByMe` для emoji-реакций чата.
- Слой сериализации публичных ответов (`apps/api/src/common/public-view.ts`): заявки, поиск, чат, профили и каталог отдаются как явные DTO и не содержат сырых ORM-полей — `moderationStatus`, счётчика `reports`, `visibility`, `privacySettings`, `role`/`isPremium`/`status`/`timezone` автора, `senderId`/`reaction.userId` и служебных временных меток. У заявки автор — `{id, username, displayName, avatarUrl, bio}`; владельческие и админ-эндпоинты сохраняют полную форму.
- Единый формат ошибок: все ответы об ошибке (`401/403/404/422/5xx`) приходят JSON-конвертом `{ ok:false, statusCode, error, message, path, method, timestamp }`; защищённые эндпоинты без токена отвечают `401` JSON, неизвестные маршруты — `404` JSON, без пустых тел.
- API отдает базовые security headers: `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, `Permissions-Policy`.
- Шапка определяет состояние входа без мигания: при логине/refresh сервер ставит читаемый hint-cookie `cofind_auth` рядом с HttpOnly-сессией, а `head`-скрипт `auth-state.js` до первой отрисовки выставляет `html[data-auth-state]`. - Realtime чат: native WebSocket `ws://localhost:4000/ws/chat` (в проде `wss://cofind2.com/ws/chat`; внешний nginx обязан проксировать `/ws/chat` с заголовками `Upgrade`/`Connection`). Для отправки сообщения нужен JWT-токен в query `?token=...` или в payload; WS-сообщения проходят те же проверки доступности пользователя, цитируемого сообщения и рисунка, что и REST.
- Realtime масштабируется горизонтально через Redis pub/sub: сообщение, созданное на любом инстансе API, публикуется в канал `cofind:chat:broadcast`, и каждый инстанс рассылает его своим WS-клиентам — можно держать несколько реплик API за балансировщиком. Подключается через `REDIS_URL`; без него сервис работает в single-process режиме, а при недоступности Redis деградирует к локальной рассылке. WS-слой имеет backpressure (медленные клиенты с переполненным буфером отключаются), heartbeat-reaping (ping/pong каждые 30с, мёртвые сокеты закрываются), per-connection flood control (≤15 сообщений/10с) и лимит соединений. `/health/ready` показывает `realtime.redis` = `connected|disconnected|disabled`.

- Форма создания заявки использует реальные теги, жанры, фандомы и персонажей из `/tags`, `/genres`, `/fandoms`, `/characters`: выбранные slug'и можно добавлять/удалять, а статус `PUBLISHED` вызывает публикацию через `/listings/:id/publish`.
- Форма новой заявки автосохраняет локальный черновик в браузере, восстанавливает его при возвращении на `/me/listings/new` и очищает после успешного сохранения.
- В форме новой заявки есть live-предпросмотр карточки: заголовок, описание, статус, рейтинг и выбранные каталоги обновляются во время ввода.
- Описание заявки, отклик, общий чат и личные сообщения используют полноценный contenteditable WYSIWYG-редактор вместо markdown-вставок в textarea: форматирование видно сразу, поддержаны жирный, курсив, зачеркнутый, списки, цитата, ссылка, undo/redo, очистка и emoji; значение сохраняется как безопасный HTML в скрытом поле, а счетчики считают видимый текст и заранее ловят слишком тяжелое форматирование.
- API дополнительно чистит rich-text для заявок, откликов, общего чата и личных сообщений: опасные теги/атрибуты вроде `script`, `onclick` и `javascript:` не сохраняются даже при прямом REST/WS-запросе.
- Web-форма заявки показывает счетчики заголовка/описания и не отправляет данные, если длина не проходит правила API: заголовок 6-140 символов, описание 20-4000 символов.
- API создания/редактирования заявки валидирует catalog slug'и до записи, принимает только approved-элементы каталога, нормализует регистр/пробелы и дедуплицирует повторяющиеся slug'и.
- Публичный `GET /listings` тоже принимает каталоговые фильтры `tag`, `genre`, `fandom`, `character` по `slug` или названию.
- Web-студия показывает все заявки автора через `/listings/mine`, позволяет искать и сортировать их, редактировать, отправлять на публикацию, закрывать, архивировать и мягко удалять.
- Web-админка для скрытых или мягко удаленных заявок показывает Restore; действие возвращает заявку в `DRAFT/PENDING`, чтобы автор мог проверить и заново отправить ее на модерацию.
- В блоке “Мои заявки” есть фильтры по статусам `DRAFT/PUBLISHED/CLOSED/ARCHIVED` с счетчиками и пустыми состояниями.
- Web-страница заявки открывается из ленты по реальному id, догружает `GET /listings/:slugOrId`, отправляет отклик через `/listings/:id/respond`, лайкает `/listings/:id/like` и подставляет id заявки в форму жалобы.
- Карточки заявок в ленте, live-блоке главной и публичном профиле содержат настоящие ссылки `/listing/:id`, а авторы карточек ведут на `/profile/:username`.
- Web-runtime нормализует все кнопки без `type` в `type="button"` и следит за динамически отрисованными кнопками, чтобы быстрые действия и карточки не отправляли формы случайно; `form method="dialog"` оставлен для нативного закрытия модалок.
- Web smoke проверяет, что каждая форма с `id` имеет явный submit-handler, а dialog-формы остаются нативными.
- Web-detail заявки показывает метрики лайков/откликов/жалоб, блокирует отклик на закрытую заявку и предлагает шаблон первого сообщения.
- Форма отклика на заявку показывает счетчик 4000 символов, блокирует пустой/слишком короткий текст и объясняет лимит 10-4000 символов до отправки.
- Связанные ссылки в detail заявки имеют реальные `/feed?...` URL, открывают ленту с фильтром по первому тегу/жанру, фандому или персонажу выбранной заявки и сохраняют фильтр в истории браузера без промежуточного `/feed`.
- Detail заявки показывает похожие заявки по общим тегам, жанрам, фандомам и персонажам и открывает их напрямую.
- Web-профиль автора открывается из детальной заявки, загружает `GET /profiles/:username`, показывает био, теги, социальные ссылки, суммарные метрики заявок, статус Premium и опубликованные заявки.
- ЛК и публичный профиль показывают расширенные предпочтения автора: стиль, грамотность, ожидаемую длину поста, темп и предпочтительный формат связи отдельным блоком.
- С публичного профиля можно открыть личный диалог через `POST /conversations/direct`; API возвращает существующий диалог между двумя пользователями или создает новый без дублей.
- `POST /conversations/direct` учитывает настройку `allowProfileMessages=false`: новый диалог с публичного профиля блокируется, но существующий диалог продолжает открываться.
- Публичные профили возвращают опубликованные approved-заявки с полными связями tags/genres/fandoms/characters и метриками `likes/responses/reports`.
- Web-публичный профиль позволяет искать по опубликованным заявкам автора, сортировать их по новизне, лайкам или откликам и листать длинные списки постранично.
- Из детальной заявки и публичного профиля можно заблокировать автора через `POST /me/blocks`; на `/me` доступен блок-лист с поиском, счетчиком и разблокировкой.
- На публичном профиле есть быстрый переход к жалобе на профиль (`PROFILE` report) с автозаполнением ID автора.
- Блокировка несуществующего пользователя возвращает 404, а не сырую ошибку базы.
- Web-inbox показывает личные диалоги, исходящие отклики, входящие отклики на ваши заявки, позволяет принять/отклонить отклик, открыть историю личного диалога и отправить сообщение.
- Открытый личный диалог формирует прямую ссылку `/me/inbox?conversation=<id>`, а прямой заход восстанавливает выбранный разговор после загрузки inbox.
- Web-inbox поддерживает фильтры `all/new/sent/dialogs` через query `tab`, поэтому можно быстро открыть новые отклики, свои отклики или только диалоги и вернуться Back/Forward.
- Web-inbox показывает сводку новых откликов, диалогов, непрочитанных и принятых откликов; список inbox поддерживает поиск/сортировку, а личный composer блокирует пустую отправку и показывает лимит 4000 символов.
- В открытом личном диалоге можно искать по автору, email и тексту сообщения, сбросить поиск из пустого состояния и скопировать прямую ссылку на разговор.
- В личном диалоге можно удалить свое сообщение; удаленные сообщения пропадают из истории.
- Список личных диалогов возвращает `unreadCount`, а web-inbox показывает бейдж непрочитанных и сбрасывает его через `/conversations/:id/read`.
- Публичный профиль `/profiles/:username` возвращает точные общие `stats` по всем опубликованным заявкам автора и `listingsPagination`; `page/pageSize`, `q` и `sort=new|popular|responses` управляют загружаемым срезом заявок, а web-маршрут использует `/profile/:username?listingsPage=N`.

## Регрессионные тесты и CI

- Регрессионные наборы: `pnpm test:unit` (юнит-тесты плюрализации через `tsx`), `pnpm test:api` (снапшот схемы публичных DTO — любое неожиданное поле валит тест, плюс пагинация и коды ошибок 401/403/404), `pnpm test:seo` (уникальные `title`/`description`, `canonical` == URL страницы включая `/profile/<username>`, `og:image`, и sitemap: нет query-параметров, есть хотя бы одна заявка и один фандом, все `lastmod` — валидный ISO; sitemap-index разворачивается в под-карты; динамическая `og.png` заявки отдаёт валидный image Content-Type) и Playwright e2e в `e2e/` (`pnpm e2e`; браузеры — `pnpm e2e:install`). Базовый URL e2e задаётся через `E2E_BASE_URL` (по умолчанию `http://localhost:8092`).
- Playwright e2e покрывают гостевые сценарии (лента → заявка → «Откликнуться», редирект `/me` → `/auth?next=/me` и возврат после входа, чтение чата без логина), видимость кнопок шапки для гостя/пользователя и полный путь регистрация → создание заявки. Поле описания заявки — rich-`contenteditable` поверх скрытой `textarea`, поэтому e2e печатает в `.rich-editor`, а успех создания проверяется по переходу на `/me` (на ошибке переход не происходит).
- CI-пайплайн `.github/workflows/ci.yml`: `build` (typecheck + сборка turbo + unit + опциональный Sentry-релиз с sourcemaps), `test` (поднимает полный стек через `deploy/docker-compose.ci.yml`, мигрирует/сидит, гоняет `test:api` + `test:seo` + Playwright) и `deploy` (по push в `main`: SSH-деплой с пробросом `SENTRY_RELEASE` и пост-деплой smoke `/health/ready`). Деплой требует repo-секретов `SSH_HOST`, `SSH_USER`, `SSH_KEY`, `DEPLOY_URL`; Sentry-релиз — опциональных `SENTRY_AUTH_TOKEN`, `SENTRY_ORG`, `SENTRY_PROJECT` (+ `SENTRY_URL` для self-hosted), без них шаг пропускается.
- Контейнер web самодостаточен и вне внешнего nginx: его конфиг проксирует общий `/api/v1/` и `/ws/` на API, поэтому клиентские вызовы SPA (логин, лента, чат) работают при standalone-запуске (CI, e2e, локально). В проде эти пути обслуживает внешний nginx до контейнера, так что блоки инертны.
- Гостевые редиректы защищённых экранов (`/me*`, `/reports/new`, `/admin`) формируют `Location` через `$http_host`, а не `$host` — порт сохраняется, поэтому редиректы корректны и на нестандартных портах (CI/e2e/локально); в проде (`cofind2.com` без порта) поведение не меняется.

Перед публичным релизом после production-сборки web:

```powershell
pnpm secrets:generate
$env:NODE_ENV="production"
$env:PUBLIC_WEB_URL="https://cofind.example.com"
$env:PUBLIC_API_BASE="https://api.cofind.example.com/api/v1"
$env:DATABASE_URL="postgresql://..."
$env:JWT_ACCESS_SECRET="<long-random-secret>"
$env:JWT_REFRESH_SECRET="<another-long-random-secret>"
$env:MEILISEARCH_HOST="https://search.example.com"
$env:MEILISEARCH_MASTER_KEY="<long-random-secret>"
$env:PAYMENT_WEBHOOK_SECRET="<long-random-secret>"
$env:MAIL_WEBHOOK_URL="https://mail.example.com/send"
$env:API_DOCS_ENABLED="false"
pnpm release:check
pnpm ux:audit
```

`release:check` падает, если в `dist` остался localhost, `dist/app.js` смотрит не на production API, не заданы HTTPS URL, слабые секреты или случайно открыт Swagger.
Перед переносом на сервер можно выполнить production web build и release gate одной командой: `pnpm release:prepare deploy/.env.production`.
`ux:audit` статически проверяет UX-контракты фронтенда: фокус, мобильные брейкпоинты, перенос длинного текста, image optimization и отсутствие viewport-scaled `font-size`.
